Application Deployment Lab

OpenShift supports high availability and load balancing of web applications. Tools such as Infinispan and JGroups provide clustering of application state in JBoss EAP. Clustering is the same whether JBoss EAP is running in the cloud, on VMs, or on bare-metal infrastructure. When your application uses web session data, the OpenShift container for JBoss EAP automatically replicates the data. To make your application distributable, you simply configure your web application to use the distributable tag.

In this lab, you replicate the HTTP session between pods in a project using a web application called session-replication.

Goal

  • Enable high availability and load balancing of a Java EE application deployed in OpenShift

1. Review session-replication Application

In this section, you download the source for the ocp-session-replication application, as in the OpenShift Logs Lab. This lab refers to this application as session-replication.

  1. Change your working directory to your home directory:

    $ cd $HOME

  2. If you have any objects in the OpenShift project from the previous labs, remove them:

    $ $HOME/ocp-appdev-labs/admin/purge_apps.sh

  3. Clone the source code of the ocp-session-replication application from this GitHub repository:

    $ cd $HOME
$ git clone https://github.com/wkulhanek/ocp-session-replication.git
$ cd ocp-session-replication
$ git checkout 3.5

  4. Examine the src/main/webapp/WEB-INF/web.xml file.

    • This standard Java EE deployment descriptor file uses the distributable tag.

    • The distributable tag is a Java EE convention indicating to the Java EE container that HTTP session state should be replicated across a cluster.

  5. Examine the src/main/webapp/index.jsp file.

    • This is a simple Java Server Pages (JSP) file. It includes Java code that defines a counter whose state is persisted in the HTTP session.

    • The counter and name of the container currently serving the HTTP request when invoked are displayed in the user’s browser.

  6. Examine the .s2i/bin/assemble file.

    • This is a custom S2I assemble script, similar to the one used in the bookstore application.

2. Review and Edit OpenShift Resources

In this section, you add customizations to the application template.

  1. Just as you did in the OpenShift Logs Lab, process the template, examine/modify the JSON source code, and create the application from the modified JSON file:

    $ cd $HOME
$ oc process openshift//eap70-basic-s2i -p APPLICATION_NAME=session-replication -p SOURCE_REPOSITORY_URL=https://github.com/wkulhanek/ocp-session-replication.git -p SOURCE_REPOSITORY_REF=3.5 -p CONTEXT_DIR= >session-replication.json

    • You use the name session-replication rather than ocp-session-replication when creating the application.

  2. Edit the generated file session-replication.json.

    • Identify the BuildConfig object and the environment variable used when a build object executes.

  3. In the BuildConfig object, note the name of the ImageStreamTag — jboss-eap70-openshift:1.4.

    • This pulls the base image of the S2I build process into the resultant application.

  4. Compare the name of this ImageStreamTag with the image streams available in the openshift namespace.

  5. Determine whether the image stream is included in the openshift namespace.

    • This particular application expects version 1.4 of the jboss-eap70-openshift image.

      To view a list of available image streams in the openshift namespace, use the oc get is -n openshift command. The image stream leveraged in the BuildConfig object of the session-replication application should match one of the image streams in the openshift namespace.

  6. In the containers section of the DeploymentConfig, verify that the following environment variables are defined for the Java EE container responsible for enabling session replication clustering:

    {
  "name": "OPENSHIFT_KUBE_PING_LABELS",
  "value": "application=session-replication"
},
{
  "name": "OPENSHIFT_KUBE_PING_NAMESPACE",
  "valueFrom": {
    "fieldRef": {
        "fieldPath": "metadata.namespace"
    }
},

  7. Exit the file.

3. Initiate Application Build and Deployment

  1. Make sure that your default service account has sufficient privileges to communicate with the Kubernetes REST API.

  2. Add the view role to serviceaccount for the project:

    $ oc policy add-role-to-user view system:serviceaccount:$(oc project -q):default

  3. Use the JSON file that you generated previously to create all of the required OpenShift objects:

    $ oc create -f session-replication.json

  4. Wait until a build is triggered.

  5. Monitor the build logs to ensure that the build is successful.

  6. Make note of the ID of this initial session-replication pod:

    $ oc get pods

  7. Monitor the log file of the newly created session-replication container to verify that the web application deployed correctly.

  8. Examine the first entry in the log file:

    Service account has sufficient permissions to view pods in kubernetes (HTTP 200). Clustering will be available.

  1. Use the oc get route command to determine the route of the session-replication web application.

  2. In a browser, navigate to the session route you just retrieved, prefixing that value with http://.

  3. Verify that your page looks similar to this:

  4. Click the Increment Counter link several times and observe that the value of the Session counter field increments by one each time.

    • The state of this counter is stored in the HTTP session. In a clustered environment, this session is replicated to other nodes.

  5. Make note of the name of the pod handling this HTTP request.

5. Initiate Clustering

5.1. Start Second Instance of Application Container

  1. Increase the number of the application’s replicas to 2:

    $ oc scale --replicas=2 dc session-replication

  2. Tail the log file of each session-replication container.

  3. Note a new log entry in the initial session-replication container similar to the following:

    21:21:45,232 INFO  [org.infinispan.remoting.transport.jgroups.JGroupsTransport] (thread-2,ee,ion-replication-1-03ssk) ISPN000094: Received new cluster view for channel server: [ion-replication-1-03ssk|1] (2) [ion-replication-1-03ssk, ion-replication-1-knn9t]
21:21:45,233 INFO  [org.infinispan.remoting.transport.jgroups.JGroupsTransport] (thread-2,ee,ion-replication-1-03ssk) ISPN000094: Received new cluster view for channel web: [ion-replication-1-03ssk|1] (2) [ion-replication-1-03ssk, ion-replication-1-knn9t]
21:21:45,234 INFO  [org.infinispan.remoting.transport.jgroups.JGroupsTransport] (thread-2,ee,ion-replication-1-03ssk) ISPN000094: Received new cluster view for channel hibernate: [ion-replication-1-03ssk|1] (2) [ion-replication-1-03ssk, ion-replication-1-knn9t]
21:21:45,234 INFO  [org.infinispan.remoting.transport.jgroups.JGroupsTransport] (thread-2,ee,ion-replication-1-03ssk) ISPN000094: Received new cluster view for channel ejb: [ion-replication-1-03ssk|1] (2) [ion-replication-1-03ssk, ion-replication-1-knn9t]

    • This log entry indicates that the JGroups/Infinispan subsystem of JBoss EAP has automatically detected the presence of the second container. A cluster is formed and the HTTP session state is replicated to the second container.

5.2. Verify Sticky Sessions

Now that a second session-replication pod is running, you verify that requests from your browser continue to be routed to the original pod.

  1. Refresh the browser page several times.

  2. Confirm that the pod serving the HTTP requests stays the same.

5.3. Verify Failover

In this section, to demonstrate session replication and failover, you delete the initial session-replication container and quickly refresh the session-replication web application’s home page. This results in the following:

  • The HTTP request is routed seamlessly to the second session-replication container.

  • The HAProxy container of OpenShift Container Platform detects that the initial session-replication container no longer exists and automatically re-routes accordingly.

  • The value of the counter stays the same.

Because the HTTP session state was previously replicated to the secondary container, this session state remains available when failure of the initial container occurs.

  1. Delete the initial session-replication container:

    $ oc delete pod <pod id of first session-replication container>

  2. Return to your browser and quickly refresh the session-replication page.

  3. Confirm that the counter value stays the same and the ID of the container serving the HTTP request has changed.

    • Because the replica number is still set to 2, OpenShift/Kubernetes quickly detects that a container failed. OpenShift then creates a replacement session-replication container.

6. Delete session-replication Objects

You explored how OpenShift handles failover through session replication.

You do not need to use the session-replication objects again, so you can delete them.

  1. Delete session-replication objects:

    $ oc delete all -l=application=session-replication
