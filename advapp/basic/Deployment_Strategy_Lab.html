Deployment Strategy Lab

The primary business driver for DevOps is agility. For an IT shop, this business requirement often translates to a requirement for continuous updates to the software and services supporting the business. This lab demonstrates the out-of-the-box deployment strategies offered by OpenShift. These deployment strategies come into play when developers update the services for which they are responsible.

Goal

  • Understand rolling and recreate deployment strategies

1. Examine deployment-example Application

This lab uses a lightweight application called deployment-example written in the Go programming language.

The Go containers for this lab are already created for you and available in Docker Hub at https://hub.docker.com/r/openshift/deployment-example/tags/. There are two tags for this container: v1 and v2. In this lab, you toggle between these two versions to trigger and observe OpenShift’s various deployment behaviors.

The lab assets for this course include the $HOME/ocp-appdev-labs/06_Deployments/deployment-example-template.json template.

2. Explore Rolling Deployment Strategy

When OpenShift is configured to use the rolling deployment strategy, updates to an application or service occur gradually across replicas. OpenShift scales up the later deployment in stages, gradually reducing the number of earlier deployments. If one of the newly deployed pods never reaches the ready state, the later deployment is rolled back and scaled down to zero.

You use this deployment strategy when your application can tolerate two versions of code running at the same time. This typically requires that your application handle N-1 compatibility—that is, data stored by the later version can be read and handled, or gracefully ignored, by the earlier version of the code. This can take many forms: data stored on disk, in a database, in a temporary cache, or in a user’s browser session. While most web applications can support rolling deployments, it is important to test and design your application to handle this strategy.

The rolling deployment strategy allows for various optional configurations:

OptionDescription

updatePeriodSeconds

Wait time between individual pod updates.

intervalSeconds

Wait time between polling deployment status after update.

timeoutSeconds

Wait time for updates before giving up.

maxUnavailable

Maximum number of pods that can be unavailable during the update. The value can be an absolute number or percentage of total pods at the start of the update.

maxSurge

Maximum number of pods that can be scheduled above the original number of pods. The value can be an absolute number or percentage of the total pods at the start of the update.

updatePercent

Percentage of replicas to scale up or down at each interval (negative value switches scale order to down/up instead of up/down).

pre

Hook executed before strategy starts deployment.

post

Hook executed after strategy finishes deployment.

In this section of the lab, you perform an OpenShift rolling update of an application across replicas.

    1. Open deployment-example in the editor:

      $ oc edit dc deployment-example

    2. Scroll down to the imageChangeParams section near the bottom of the file.

    3. Change the name of the ImageStreamTag to deployment-example-is:v2:

3. Explore Recreate Deployment Strategy

When OpenShift is configured to use the recreate deployment strategy, updates to an application or service occur in discrete stages. OpenShift scales all replicas of an application down to zero, then scales these replicas back up to full.

You use the recreate deployment strategy when your application cannot tolerate two versions of code running simultaneously. In this section, you watch OpenShift update an application using the recreate deployment strategy.

First, you modify the application’s deployment configuration to change the deployment strategy.

  1. Open the deployment-example deployment configuration file:

    $ oc edit dc deployment-example

  2. Scroll down to the strategy section and change type to Recreate:

  3. Scroll down to the imageChangeParams section near the bottom of the file.

  4. Change the name of the ImageStreamTag back to deployment-example-is:v1.

  5. Save the change and exit.

  6. Trigger a new deployment:

    $ oc rollout latest dc/deployment-example

  7. In the OpenShift web console, browse to the list of pods supporting the deployment-example application.

    • Within a minute of changing the version of the Docker image used by the application, the pod listing begins to change at five-second increments:

    • All of the old pods terminate simultaneously.

    • Three new pods running the v1 version of the Docker image are spun up at the same time.

  8. Refresh your browser window to verify that OpenShift reverts to using v1 of the image.

4. Delete deployment-example Objects

You have seen how OpenShift handles rolling and recreate deployments. You no longer need the deployment-example objects for subsequent exercises, so you can delete them.

  1. Delete the deployment-example objects:

    $ oc delete all -l=app=deployment-example
