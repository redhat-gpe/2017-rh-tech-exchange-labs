Java EE Application Deployment Lab

In this lab, you create a containerized Java EE application using OpenShift’s S2I workflow. Instead of using the OpenShift web console, you use the oc client utility. Leveraging the S2I process, you build a Java EE bookstore application layered into a JBoss Enterprise Application Platform (JBoss EAP) container.

Goal

  • Use OpenShift S2I workflow to manage the life cycle of a Java EE application

Prerequisites

  • OpenShift Container Platform oc command line client

  • Git client installed on client workstation

  • Completion of previous labs using the web console

In previous labs, you installed the OpenShift oc client utility on your client workstation. At that time, the oc client created a valid authentication token with the OpenShift master API. If that token has expired, re-authenticate using oc login. You receive notification of an expired token when executing any oc command that invokes the remote OpenShift master API.

1.1. Explore Route URL

Business applications and services deployed in OpenShift often need to be accessible from external clients. In this lab, for example, you need to access the deployed bookstore application from your local browser. In OpenShift, this is done by defining a route.

Your application’s route is unique to your project, such as:

http://bookstore-wkocpappdev.apps.na1.openshift.opentlc.com/

The following list describes the segments of this route URL:

  • bookstore: Name of application

  • wkocpappdev: Name of your OpenShift project

  • apps.na1.openshift.opentlc.com: DNS domain of the OpenShift environment, provided in your lab registration email

2. Deploy Java EE Application Using S2I

2.1. Clone application-templates Project

Red Hat produces OpenShift Container Platform templates and image streams for Red Hat JBoss Middleware, which are used in this lab. In this section, you review these artifacts to become familiar with how to create your own.

  1. At the command prompt of your client workstation, clone the application-templates repository:

    $ cd $HOME
$ git clone https://github.com/jboss-openshift/application-templates

    • The application-templates/README.md file provides an overview of how to use the OpenShift Container Platform xPaaS application templates contained in this project.

    • The application-templates/jboss-image-streams.json defines the JBoss-related image streams used as part of an OpenShift S2I workflow.

  2. Review the application-templates/eap directory and note that it includes templates for various JBoss EAP deployment topologies in an OpenShift environment.

    • In this lab, the application-templates/eap/eap64-mysql-s2i.json template is used to easily support automated deployment of a Java EE web application to a JBoss EAP container.

    • The Java EE web application connects to a mysql-server that is local to the Java EE container.

      Containers in OpenShift are immutable and ephemeral. Any state that may have accumulated during the life span of a container is lost when that container is stopped. These immutable and ephemeral characteristics of containers apply even to database containers (for example, the mysql-server container used in this lab). It is often the case that database containers are configured with a persistent volume to allow for durable storage of data while the containers themselves are stopped and replaced. (The next lab provides more information on setting up persistent volumes.) For the purpose of this lab, keep in mind that the mysql-server container is ephemeral and therefore database state is lost when the container is stopped.

2.2. Create OpenShift Service Accounts and Secrets

When a person uses the OpenShift CLI or web console, an API token authenticates the user to the OpenShift Container Platform API. However, when a regular user’s credentials are not available, it is common for components to make API calls independently. For example:

  • Replication controllers make API calls to create or delete pods.

  • Applications inside containers can make API calls for discovery purposes.

  • External applications can make API calls for monitoring or integration purposes.

Service accounts provide a flexible way to control API access without sharing a regular user’s credentials.

  1. Container images used by this template require secrets for configuring HTTPS, review the project resources defined in the following data file:

    $ cat application-templates/secrets/eap-app-secret.json

  2. Note the two resources:

    • service account: eap-service-account

    • secret: eap-app-secret

  3. Add the service account and secret to the project:

    $ oc create -f application-templates/secrets/eap-app-secret.json

2.3. Define OpenShift Bookstore Resources

In this section of the lab, you generate OpenShift objects specific to the bookstore application.

  1. Review the contents of the eap64-mysql-s2i.json template file:

    $ more application-templates/eap/eap64-mysql-s2i.json

  2. Use the eap64-mysql-s2i.json configuration file as a base template to create a list of OpenShift bookstore resources (in JSON format) and pipe the output to a /tmp/bookstore.json file:

    oc process -f application-templates/eap/eap64-mysql-s2i.json -p APPLICATION_NAME=bookstore -p SOURCE_REPOSITORY_URL=https://github.com/wkulhanek/ocp-appdev-bookstore.git -p SOURCE_REPOSITORY_REF=master -p CONTEXT_DIR= -p DB_JNDI=java:/jboss/datasources/mysql -p DB_USERNAME=bookstore -p DB_PASSWORD=jb0ssredhat -p DB_DATABASE=bookstore > $HOME/bookstore.json

  3. Search for the BuildConfig string in the $HOME/bookstore.json file, scroll down to the sourceStrategy section, review it and change it to look like the following:

    Be sure you change the version of the ImageSreamTag and add the "env" section, and make sure you add all the right commas!
    "strategy": {
		 "sourceStrategy": {
				 "env": [
						 {
								 "name": "MAVEN_MIRROR_URL",
								 "value": ""
						 },
						 {
								 "name": "MAVEN_ARGS",
								 "value": "clean package -DskipTests -s config/nexus-hardcoded-settings.xml"
						 },
						 {
								 "name": "ARTIFACT_DIR",
								 "value": ""
						 }
				 ],
				 "forcePull": true,
				 "from": {
						 "kind": "ImageStreamTag",
						 "name": "jboss-eap64-openshift:1.4",
						 "namespace": "openshift"
				 }
		 },
		 "type": "Source"
 },

  4. Save the updated file and quit your editor.

2.4. Build Bookstore Application

Now that you have defined resources in the $HOME/bookstore.json file, you create those resources in OpenShift. In this section of the lab, you see OpenShift’s S2I capabilities in action.

  1. Create the resources:

    $ oc create -f $HOME/bookstore.json
service "bookstore" created
service "secure-bookstore" created
service "bookstore-mysql" created
route "bookstore" created
route "secure-bookstore" created
imagestream "bookstore" created
buildconfig "bookstore" created
deploymentconfig "bookstore" created
deploymentconfig "bookstore-mysql" created

    • OpenShift/Kubernetes automatically triggers execution of the build configuration.

  2. Begin by tailing the project event log:

    oc get events -w

    • Within about a minute after creating the bookstore-related OpenShift resources, the OpenShift Kubernetes scheduler starts a variety of events.

  3. Examine the bookstore events when they appear in the event log.

  4. Press CTRL+C to exit from the log.

  5. Note the existence of S2I build resources:

2.5. Deploy Bookstore Application

The ReplicationConfig object of the bookstore application by default sets replica = 1, and OpenShift/Kubernetes automatically creates a bookstore container. Immediately after the bookstore application is built, the S2I workflow adds the built web artifact into a thin Docker image layered onto the JBoss EAP image. A container of this JBoss EAP image starts.

The JBoss EAP container is configured so that its server log is made available using OpenShift client tools.

  1. View the JBoss server.log.

  2. Determine the name of the pod in which the JBoss EAP container resides:

    $ oc get pod
NAME                      READY     STATUS      RESTARTS   AGE
bookstore-1-build         0/1       Completed   0          15m
bookstore-1-deploy        1/1       Running     0          4m
bookstore-1-mul6e         0/1       Running     0          3m
bookstore-mysql-1-5ef8i   1/1       Running     0          15m

  3. Identify the bookstore-1-* pod (mul6e in the example above).

    • The log may indicate 0/1 under READY if it is still starting up.

  4. Using the pod name identified in the previous step, tail the JBoss server log:

    $ oc logs -f <pod_id>

  5. Wait for the startup to finish—this can take a few minutes.

    • Expect to see output in your log similar to the following:

      18:28:34,809 INFO  [org.jboss.as] (Controller Boot Thread) JBAS015961: Http management interface listening on http://127.0.0.1:9990/management
18:28:34,810 INFO  [org.jboss.as] (Controller Boot Thread) JBAS015951: Admin console listening on http://127.0.0.1:9990
18:28:34,810 INFO  [org.jboss.as] (Controller Boot Thread) JBAS015874: JBoss EAP 6.4.14.GA (AS 7.5.14.Final-redhat-2) started in 154779ms - Started 432 of 524 services (129 services are lazy, passive or on-demand)
      If your bookstore application fails to start it is most likely because the liveliness or readiness Probes kill it before it successfully launched. You can see that this is happening by using the oc get events command. If this happens go to your web console, find the deployment configuration for the bookstore, and under Actions select Edit Health Checks. Then add a value of 240 for the Initial Delay under both the Readiness Probe and Liveliness Probe settings.

Your bookstore web application is successfully built and deployed using OpenShift’s S2I workflow. Now that it is started, you can authenticate to it and peruse its functionality.

  1. If you do not recall the route to your bookstore application, execute the oc get routes command:

    $ oc get routes

    http://bookstore-wkocpappdev.apps.${REGION}.openshift.opentlc.com/bookstore
