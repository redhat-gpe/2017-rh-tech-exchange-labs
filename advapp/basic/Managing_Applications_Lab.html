Managing Applications Lab

In previous labs, you explored OpenShift resources using the OpenShift client tools. The OpenShift CLI tool has an extensive feature set and is the primary tool for interacting with OpenShift. This lab introduces additional management capabilities of OpenShift.

Goal

  • Manage applications deployed in OpenShift using advanced client tools

1. Explore OpenShift Resources Using oc Client

1.1. Create Application

For the purpose of this lab, you create a lightweight Node.js application in OpenShift.

  1. Make sure that you are in the project you created before (e.g. wkocpappdev):

    oc project wkocpappdev

  2. Create the application from a GitHub repository:

    oc new-app https://github.com/openshift/nodejs-ex.git

    • Expect your output to look similar to this:

1.3. View Details and Associated Events

You can use the oc describe <resource type> <resource name> command to get a concise description of the details of any OpenShift resource.

  1. View the details of an OpenShift deployment configuration resource named nodejs-ex:

    oc describe dc nodejs-ex

  2. Review the details of the other nodejs-ex resources and note that for some the oc describe command lists the associated event history.

1.4. View Resource Attributes

The OpenShift CLI tool also allows for a complete view of the current state of a resource using the oc get <resource type> <resource name> -o yaml syntax. Examining the associated attributes of each resource type allows you to manage and troubleshoot your deployed applications.

  1. Get an exhaustive view of the attributes of the nodejs-ex deployment configuration resource:

    oc get dc nodejs-ex -o yaml

  2. Review the attributes.

2. Use Replicas to Control Application Life Cycle

Kubernetes ensures that your application is correctly deployed for the number of replicas defined in its associated replicationcontroller resource. By defining the number of replicas, new Linux containers can be spun up or down to support your application.

You use the oc scale command to define the number of replicas:

oc scale --replicas=<replica count> dc <deployment configuration name>

  1. For the OpenShift deployment configuration named nodejs-ex, specify that two replicas are always running:

    oc scale --replicas=2 dc nodejs-ex

  2. Verify that there are now two pods running for nodejs-ex:

    $ oc get dc nodejs-ex
NAME        REVISION   DESIRED   CURRENT   TRIGGERED BY
nodejs-ex   1          2         2         config,image(nodejs-ex:latest)

  3. For the same nodejs-ex deployment configuration, stop all running replicas:

    oc scale --replicas=0 dc nodejs-ex

  4. Verify that there are now no pods running.

  5. Revert to one replica of your nodejs-ex application.

3. View Events and Logs

3.1. View Project Events

Viewing the OpenShift events associated with your project allows you to see what your applications are doing. In particular, if a build or deployment of your application fails, project events provide critical hints to the failure’s cause. Red Hat recommends as a best practice that you "tail" your project events when the application goes through its build and deployment life cycles.

  1. To continuously tail the project’s events, open a new terminal window and execute:

    oc get events -w

  2. In your original terminal window, manipulate your application—for example, create it, start the build, and change the replica count.

    If a failure occurs during the life cycle of your application, refer to the event log and identify messages that can assist you in troubleshooting the problem.

  3. Hit Ctrl-C to exit from watching events.

3.2. Build Logs

For the applications using OpenShift’s S2I feature, it is often critical to view the log files generated during an application’s build. The build log contains crucial information, such as the output of a Maven build, if using a Java-based application. You can view build logs as follows:

  1. Once a build of your application is triggered, either automatically by OpenShift or manually using oc start-build, identify the name of the build:

    $ oc get build
NAME                    TYPE      FROM          STATUS     STARTED          DURATION
nodejs-ex-1             Source    Git@9457131   Complete   8 minutes ago    1m42s

  2. Using the pod name of the build object, begin to tail the build log:

    oc logs -f build/nodejs-ex-1
    Build logs are associated with the build pod. As long as the build pod still exists, the build logs remain available for review even after the build is complete.

3.3. View Application Logs

This section provides instructions on how to view log files associated with your running, containerized application managed by OpenShift. Once an application pod is running, you can tail its log files.

  1. Identify the name of the running application pod:

    oc get pod
NAME                          READY     STATUS      RESTARTS   AGE
nodejs-ex-1-build             0/1       Completed   0          10m
nodejs-ex-1-x1n9v             1/1       Running     0          37s

  2. Using the name of the pod, tail the application containers log file:

    oc logs -f nodejs-ex-1-x1n9v

4. Execute Command in Container

OpenShift can execute remote commands in a pod using the oc client tool. This is especially helpful in troubleshooting a running application. Arbitrary commands can be executed, as long as they are available within the container and pods in which you execute the remote commands. This ability is useful for diagnosing files, contents, and processes from within the container itself. Alternatively, the oc rsh command also accepts additional arguments that enable the use of an interactive console. This is useful for more in-depth troubleshooting sessions.

  1. Execute a command in a running container using the following oc utility syntax:

    oc rsh <pod_ID> <command>

  2. Identify the pod and container names running your application.

    These are the same steps you executed earlier when tailing the log file of your running application.

    For example, to view the DNS configuration file of the base operating system in the nodejs-ex container, use:

    oc rsh nodejs-ex-1-x1n9v cat /etc/resolv.conf

  3. Launch a remote shell in the same nodejs-ex pod:

    $ oc rsh nodejs-ex-1-x1n9v
sh-4.2$

  4. Type exit to leave the remote shell and return to your local environment’s shell.

    Access to containers in your pod via the oc rsh command provides root access. Use the command with caution—a misstep can corrupt your container, forcing a replacement container to spin up.

5. Change OpenShift Resource

5.1. Edit Resource Attributes

OpenShift allows you to edit resource attributes. Changes to the resource attributes are immediately persisted, and OpenShift automatically makes runtime adjustments as needed. In this section of the lab, you change the number of pods in the deployment configuration of your existing nodejs-ex application.

  1. View the details of the deployment configuration resource associated with the nodejs-ex application:

    $ oc get dc nodejs-ex
NAME        REVISION   DESIRED   CURRENT   TRIGGERED BY
nodejs-ex   1          1         1         config,image(nodejs-ex:latest)

  2. From the terminal window, edit the deployment configuration:

    oc edit dc nodejs-ex

  3. Increase the number of replicas; for example, if it is currently 1, then change it to 2:

    spec:
  replicas: 2

  4. Save the change.

  5. Verify that current number of replicas is now 2:

    $ oc get dc nodejs-ex
NAME        REVISION   DESIRED   CURRENT   TRIGGERED BY
nodejs-ex   1          2         2         config,image(nodejs-ex:latest)

5.2. Delete Resources Associated with Label

When you initially create the nodejs-ex application, OpenShift creates a label named app=nodejs-ex and assigns it to all of the resources associated with the application. This exercise shows you how to selectively delete just the resources associated with nodejs-ex.

  1. Display only resources associated with the nodejs-ex application:

    oc get all -l app=nodejs-ex

  2. Delete only resources associated with the nodejs-ex application:

    oc delete all -l app=nodejs-ex
