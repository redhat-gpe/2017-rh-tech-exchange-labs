1. Redeploying OpenShift Certificate Authorities

1.1. Configuring OpenShift to use an Intermediary CA

  1. Background

MitziCom has their own Public Key Infrastructure to provide keys and certificates to secure communications. The PKI administrators have provided an intermediary signing key and certificate dedicated to the OpenShift project. As the OpenShift administrator, use the openshift-ansible deployment and management system to configure OpenShift to use this intermediary key and cert to secure OpenShift’s communications. The openshift-ansible system will use the intermediary certificate to generate all the certificates necessary to run OpenShift. The openshift-ansible system can also be used to update certificates in a running OpenShift system.

Conveniently, the MitziCom administrators put the intermediary CA cert and key for you to download on a secret web server. Download them onto your bastion host.

1.2. Preparing Your Environment

  1. Login to your OpenShift Bastion Host and switch to the root user.

  2. In this High Availability environment, the OpenShift Master API service is accessed through the Load Balancer. Find the proper hostname and port of the Master API from the ansible inventory file openshift_master_cluster_hostname and openshift_master_api_port, and put it in a shell variable for ease of use.

    MASTER_API_PORT=$(sed -n 's/openshift_master_api_port=//p' /etc/ansible/hosts || echo 8443)
MASTER_API_HOSTPORT=$(sed -n 's/openshift_master_cluster_hostname=//p' /etc/ansible/hosts):${MASTER_API_PORT}

  3. Do the same for the Router on the first infrastructure node. Note that the port number will always be 443.

    ROUTER=$(perl -ne 's/^(infranode1\..*?\.internal).*$/$1/ and print and last' /etc/ansible/hosts):443

  4. Retrieve the new "Intermediary CA" keys from the following location.

    wget http://admin.shared.example.opentlc.com/certs/intermediary.cert.pem -O /root/ca.crt
wget http://admin.shared.example.opentlc.com/certs/intermediary.key.pem -O /root/ca.key

1.3. Examine the Existing Certificates

  1. Start by examining the OpenShift Platform CA certificates on the master server with ansible and the openssl tools. This CA cert is known as the "OpenShift Platform CA," because it’s used to sign most of the certs critical to the OpenShift system. Note the "Subject" is "/CN=openshift-signer@" followed by a Unix epoch timestamp. That’s the CA certificate that’s created by default by the openshift-ansible deployer with the OpenShift oc adm ca commands when no other configuration is found. It’s considered a "self-signed" certificate, because there’s no other CA cert that has signed it. We’ll be replacing the Platform CA’s self-signed cert with the intermediary certificate, so that a common, externally generated CA can validate it and the certificates it signs.

    ansible masters -m shell -a 'openssl x509 -text -in /etc/origin/master/ca.crt | grep Subject:'

1.4. Deploy New Platform CA Files

  1. Edit your ansible inventory file section [OSEv3:vars] to include the new CA cert and key.

    openshift_master_ca_certificate={'certfile': '/root/ca.crt', 'keyfile': '/root/ca.key'}
#
# NOTE: CA certificate will not be replaced with existing clusters.
# This option may only be specified when creating a new cluster or
# when redeploying cluster certificates with the redeploy-certificates
# playbook.

  2. Run the redeploy-openshift-ca.yml ansible playbook to apply the new Intermediary CA cert and key to the system and regenerate the necessary dependent certs and keys. You may watch the ca_output log closely for success. The update process takes about 12 minutes.

    time ansible-playbook -f 20 /usr/share/ansible/openshift-ansible/playbooks/byo/openshift-cluster/redeploy-openshift-ca.yml 2>&1 > ca_output

2. Exploring OpenShift CA Certificates

In this section of the lab, we will validate the various certificates that have been generated by the openshift-ansible installer and the OpenShift system. The primary goal of this section to familiarize the student with the various security contexts implied by the certificate and key generation. The student will gain proficiency with ansible and openssl, and will gain skills in debugging certificates and TLS in the OpenShift environment.

2.1. Understand the Various CA Certificates in the Master Configuration

  1. Which certs and keys are created when you update the CA cert with ansible? Write a short script to examine all the certs in the master configuration directory to determine the subject and issuer of all the certificates.

    ansible masters -m shell -a 'for x in $(ls /etc/origin/master/*.crt); do \
echo $x; openssl x509 -subject -issuer -in ${x} -noout; echo; done' | less

  2. You should see most, but not all of the certs on the Master configuration have been signed by the issuer of the certificate we installed in the previous section:

    "issuer= /C=US/ST=North Carolina/O=Red Hat, Inc./OU=GPTE DevOps/CN=Red Hat OpenTLC Classroom Intermediary CA/emailAddress=gpte-devops-automation@redhat.com"

  3. You will see some certificates that are not signed this way - they are associated with the "etcd" and "service-signer" components of the system. The appear as follows:

    /etc/origin/master/master.etcd-ca.crt
subject= /CN=etcd-signer@1503514452
issuer= /CN=etcd-signer@1503514452
Note
 There are different security contexts of those certs singed by the CA we provided, versus the certs that are signed by a different CA.
Note
 Also note that the .kube directory has been modified on your masters. The ansible playbook has updated the certificates necessary for the oc command to function.

2.2. Understanding the Various CA Certificates in the Node Configurations

  1. The nodes of the OpenShift system must also have access to certificates. Note the difference between these certificates and those on the Masters.

    # ansible nodes -m shell -a 'for x in $(ls /etc/origin/node/*.crt); do \
echo $x; openssl x509 -subject -issuer -in ${x} -noout; echo;done' | less
    Note
    		 Note that all of the certs associated with the Node processes are signed by the CA cert we introduced. Why might this be so? Where are the certs for the routers?

3. Recap

When redeploying the OpenShift Platform CA, it only affects communication between the masters, nodes, and docker. It allows etcd, router, and registry to continue unchanged. This strategy:

  1. enhances the separation of concerns between platform administration and the rest of the components, protecting from privilege escalation.

  2. allows the administrator to plan with the various teams separately for the very minor outages for the certificate replacement to occur in an orderly fashion. :scrollbar: :data-uri: :imagesdir: images :toc2: :opencf: OPENTLC lab portal :course_name: Red Hat OpenShift Operations :account_management: OPENTLC Account Management page :catalog_name: OPENTLC Cloud Infrastructure :open_shared_ocp: OPENTLC OpenShift portal :need_client: false :show_solution: true

4. Redeploying Component Certificates

4.1. Introduction

Components of the OpenShift system have their own Certificate Authorities. In the previous lab we updated our Platform CA with our Intermediary CA certificate. The system continued to function because deploying the new CA only affected which client certificates would be accepted by the master services, node services, and docker.

Let us now redeploy the certificates of the rest of the OpenShift cluster. New certificates will be generated each from their own CA certificates. Some CA certificates have not changed, while others have. We’ll examine that in this lab.

Finally, we’ll import the root certificate into our browsers, and we’ll have access to the web console without having to accept error messages and accept self-signed certificates, just like you would at an organization that ran their own CA.

  1. In Chrome, go to the following URL: chrome://settings/?search=Manage%20certificates and click "Manage Certificates."

  2. Your operating system’s certificate management box should come up, and you’ll need to add the Root CA cert there.

    • On Macintosh, import the Root CA file into the "System" keychain.

    • Click the "System" keychain, then click on the padlock above where it says "Click to unlock the System keychain," enter your password, and click "Modify Keychain."

    • Click the "+" at the bottom of the box to add the certificate.

    • Select the name of the new Root CA from the list, and hit command and I to "Get Info" on the cert.

    • Expand the "Trust" section by clicking the triangle, and select the dropdown box next to "Secure Sockets Layer (SSL)" and set it to "Always Trust". As before, you may be prompted to enter your password again. Do so. Then, close the Keychain Access window.

  3. Browse to your loadbalancer’s public hostname https://loadbalancer.$GUID.example.opentlc.com/ and note that you do not get a certificate error, you are able to login and browse without warnings or errors.

5. Default (Wildcard) Router Certificates Lab

5.1. Background

OpenShift’s Router may be configured with a default wildcard certificate. It can be used to provide a convenient way for applications deployed to the platform to take advantage of some level of out of the box encryption without being required to use their own custom certificates. This is generally a recommended practice, at least in a Non-Production scenario to encourage exploration, experimentation, and rapid development.

In order to configure a default wildcard certificate, a certificate must be provisioned that is valid for *.<app domain>, where <app domain> is the value of openshift_master_default_subdomain. Once provisioned, you will need to place your cert, key and ca cert files on your ansible host, and add the following line to your ansible inventory.

openshift_hosted_router_certificate={"certfile": "/path/to/apps.c1-ocp.myorg.com.crt", "keyfile": "/path/to/apps.c1-ocp.myorg.com.key", "cafile": "/path/to/apps.c1-ocp.myorg.com.ca.crt"}

6. Custom (Named) Certificates Lab

Custom (Named) certificates are now reliably managed by ansible in OpenShift 3.6. They are meant of offer access to the public hostnames of the Master API server and web console. Multiple named certificates can be used for multiple hostnames for the public hostnames of the Master API. The ansible variable openshift_master_named_certificates is additive for each run of ansible. A special second variable, openshift_master_overwrite_named_certificates can be used to replace previous named certificates with the currently configured named certificate.

6.1. Validate DNS and Existing Named Certs

Note
 In this example, we’ll be using your loadbalancer’s public hostname for the named certificate, as that’s what has been automatically setup by our system. Test to make sure it’s working in DNS by executing the following:

  1. Log into your OpenShift Bastion Host

  2. Set an environment variable to equal your environment’s GUID.

    export GUID=<your GUID>

  3. Set an environment variable to equal the hostname of the new Named Certificate

    NAMED_CERT=loadbalancer.${GUID}.example.opentlc.com

  4. Exeute the following command to ensure correct DNS.

    host ${NAMED_CERT} && echo "DNS for ${NAMED_CERT} is OK."

  5. The output should resemble the following.

    loadbalancer.alb1.example.opentlc.com has address 34.233.202.129
DNS for loadbalancer.alb1.example.opentlc.com is OK.

  6. Ensure named cert doesn’t already exist in OpenShift cluster.

    1. ssh to the OpenShift master1.${GUID}.internal host and examine the named certificates configuration setting.

      [ec2-user@master1 ~]$ sudo ls /etc/origin/master/named_certificates/* || echo Empty
ls: cannot access /etc/origin/master/named_certificates/*: No such file or directory
Empty
[ec2-user@master1 ~]$ exit

6.2. Install the Named Certificate and Update the Inventory File

  1. Get the named certificate, and place it in /root/ on your OpenShift Bastion Host.

    wget http://admin.shared.example.opentlc.com/certs/loadbalancer.alb1.example.opentlc.com.cert.pem
wget http://admin.shared.example.opentlc.com/certs/loadbalancer.alb1.example.opentlc.com.key.pem

  2. Edit the inventory file to include the named certificate.

    1. Find the following section of your ansible inventory

      openshift_master_cluster_method=native
openshift_master_cluster_hostname=loadbalancer1.alb1.internal
openshift_master_cluster_public_hostname=loadbalancer.alb1.example.opentlc.com

    2. Append the follwing lines directly beneath the above

      openshift_master_named_certificates=[{"certfile": "/root/loadbalancer.${GUID}.example.opentlc.com.cert.pem", "keyfile": "/root/loadbalancer.${GUID}.example.opentlc.com.key.pem"}, "names": ["loadbalancer.${GUID}.example.opentlc.com"]}]
openshift_master_overwrite_named_certificates=true

    3. Save the changes to the inventory file.

6.3. Deploy and Validate the Named Certificate

  1. Deploy named cert.

    ansible-playbook -f 20 /usr/share/ansible/openshift-ansible/... TODO

  2. Validate deployed named cert.
